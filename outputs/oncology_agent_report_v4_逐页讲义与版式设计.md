# 肿瘤学 Agent 汇报：PPT 制作讲义

目标：面向医学工作者介绍“智能体与肿瘤学结合的新方向”，重点是拓展视野、建立正确边界、展示真实研究信号。页面文字尽量少，每页只讲一个核心观点；所有场景与案例均绑定真实文献或公开数据来源。

成稿结构：15 页。第 12-13 页用于衔接网站演示；第 14 页总结，第 15 页未来展望；不设置单独参考文献页，改为每页页脚小字来源。

---

## 总体叙事

### 主线

1. 肿瘤学的问题不是“缺一个会聊天的模型”，而是“证据来源太多、流程太长、需要可追踪整合”。
2. Agent 的价值在于把 LLM 放进受约束的证据工作流：检索、工具、结构化输出、人工复核。
3. 文献显示：裸 LLM 直接做治疗建议风险较高；RAG、工具调用和任务拆解后的表现更有希望。
4. 近期最适合讲给医学听众的方向：证据检索、试验匹配、分子报告解释、MDT 会前准备、多模态协作。
5. 真正演示时只展示一个低风险网站工作流：Onco-MDT Evidence Brief，并用两页完成“为什么演示它”和“它如何运行”的过渡。

### 视觉原则

- 风格定位：国际医学学术会议风，接近 Nature / NEJM / JCO 风格的清爽学术汇报；关键词是 clean academic、evidence-first、medical conference、diagram-driven。
- 整体气质：克制、可信、专业、开放视野；避免科技发布会、互联网产品发布会或科幻医疗风。
- 背景：优先使用白色或极浅冷灰背景，少用大面积深色背景；保持充足留白，让流程图和证据数字更容易被专家快速扫读。
- 配色：主色使用深海军蓝或深灰蓝，强调色使用青绿色/蓝绿色；少用紫色、霓虹蓝、赛博渐变和高饱和发光效果。
- 字体层级：标题清楚、短句化；页面正文少，尽量由图示承担解释功能。标题建议深蓝大字，正文深灰，来源脚注浅灰小字。
- 每页屏幕文字控制在 30-60 字内，详细解释放到讲述文案。
- 每个场景页不做大段文字，改成“简化论文图 + 1 个关键数字角标 + 1 句临床转译”。
- 第 5 页只做方向地图，标注对应论文，不直接展示准确率、F1、百分比等数字，避免把不同研究指标误读为横向比较。
- 不放单独参考文献页；每页底部放 1 行来源，例如：`Source: TrialGPT, Nat Commun 2024; TrialMatchAI, Nat Commun 2026`。
- 图片/图示尽量是简化流程图、证据路径图、矩阵图，不使用复杂论文原图截图。可以参考论文图，但重新绘制成医学听众能一眼理解的版本。
- 图像选择：优先使用医学流程、证据链、论文/指南/数据库、MDT 讨论、试验筛选、分子报告等抽象图示。避免夸张 AI 大脑、全息医生、机器人、发光神经网络、暗色控制台和过度拟人化 AI。
- 数据呈现：每页最多 1-3 个关键数字，数字作为证据支撑，不要做密集表格或 dashboard。第 6 页的数字只作为小证据条，不做大号 KPI。
- 国际会议适配：如果面向国际会议，页面主标题和关键标签可使用英文或中英双语；讲述语言可保持中文。页脚来源建议保留英文期刊名和年份。
- 生成图片要求：若使用 gpt-image-2 生成场景图，提示词应明确要求 white/light-gray academic medical slide style、deep navy title、teal accents、no readable fake text、no logos、no sci-fi hologram、no cyberpunk glow。

## 第 2 页：为什么肿瘤学特别需要“证据整合”

### 页面任务

说明肿瘤学天然适合 Agent：信息源多、更新快、跨学科、依赖证据链。

### 建议标题

肿瘤学难点：不是一个答案，而是一条证据链

### 屏幕内容

中心词：`一个临床问题`

周围 7 个来源：

- 病历与治疗史
- 影像
- 病理
- 分子检测
- 指南
- 文献
- 临床试验

底部一句话：`Agent 的价值：把分散信息整理成可复核的证据链`

### 图与排版

- 用“中心放临床问题、外围放信息源”的放射图。
- 中心可以是一个病例问题气泡，例如：`进展后下一步 MDT 应讨论什么？`
- 外围图标用医学图标，不要放长文。
- 右下角放小标签：`Human-in-the-loop`。

### 讲述文案

肿瘤学的复杂性，很多时候不在于某一个问题本身有多难，而在于证据来源非常分散。

一个病例进入讨论时，我们往往要同时看既往治疗、影像进展、病理类型、NGS 或 IHC 结果、指南推荐、最新论文以及临床试验。也就是说，医生面对的不是一个孤立答案，而是一条需要不断核对的证据链。

所以在这个场景下，Agent 的价值不是替医生做决定，而是把这些分散信息整理起来，让医生和 MDT 更快看到：已有依据是什么、还缺什么信息、哪些结论需要进一步复核。

### 数据来源

- Benary et al., JAMA Network Open 2023：精准肿瘤学解释需要人工检索研究和数据库，复杂变异超出指南时尤其困难。
- TrialGPT, Nature Communications 2024：患者-试验匹配需要分析病史和复杂入排标准，人工流程耗时且易出错。

### 建议标题

从 Chatbot 到 Agent：从回答问题到完成流程

3. `Agent：证据工作流`

### 讲述文案

这里先做一个概念区分。

LLM 本身主要负责生成文本，Chatbot 是把这种生成能力包装成对话界面。而 Agent 更进一步，它不是只回答一句话，而是把模型放进一个流程里：先拆解任务，再检索资料，必要时调用数据库或工具，最后按固定格式输出结果。

因此在医学场景里，我更倾向于把 Agent 理解为一种受约束的证据整理流程，而不是一个更像医生的聊天框。

## 第 4 页：医学 Agent 的安全底座

### 页面任务

建立全场安全边界：先检索，再生成；先引用，再解释；最后人工复核。

### 建议标题

医学 Agent 的底线：让生成被证据约束

### 屏幕内容

主流程：

`临床问题 → 证据检索 → 工具核验 → 结构化摘要 → 医生复核`

每个节点下只放 1 个词：

- 指南 / PubMed / 试验 / 数据库 / MDT

右侧放一个红色小框：

`不适合：直接输出诊断或治疗决策`

### 图与排版

- 横向流程图，5 个节点。
- 上方主流程，右下角放“不能做什么”的风险框。
- 这页可以成为后面所有场景的共同底层逻辑。

### 讲述文案

医学场景里最重要的不是模型说得是否流畅，而是它能不能把回答绑定到可靠来源。

一个相对安全的医学 Agent，应该先检索证据，再做归纳；先给出引用，再解释意义；最后把不确定性和待复核项交给医生。这样设计并不是说可以完全消灭风险，而是让风险能够被看见、被追踪、被复核。

所以后面所有应用场景，我都会围绕这条底线来讲：先证据，后生成；先引用，后解释；最终由医生复核。

### 数据来源

- Almanac, NEJM AI 2024：RAG 框架在 ClinicalQA 中正确可信引用约 91.11%。
- Hao et al., npj Digital Medicine 2025：癌症决策相关 LLM 研究平均准确率 76.2%，诊断任务平均 67.4%，提示不能直接自动决策。

## 第 7 页：场景一，指南与证据检索

### 页面任务

用一个简化 RAG 图讲清：指南/证据检索是最适合先落地的低风险场景之一。

### 建议标题

场景一：指南与证据检索

### 标题下小字

这是一个证据检索型 Agent 工作流：将临床问题转化为可追踪来源的证据摘要，辅助医生快速定位指南、文献和适用条件。

### 屏幕内容

主流程：

`临床问题 → 高质量来源 → 检索片段 → 带引用摘要 → 待复核项`

右上角一个研究数据转译卡：

- 大字：`Almanac 临床问答中约 91% 的引用正确可信`
- 小字：提示 RAG 更适合生成可核验的 Evidence Brief，而不是依赖模型记忆。

底部临床转译：

`适合：指南更新、证据摘要、MDT 问题准备`

### 图与排版

- 参考 Almanac 图 1，但大幅简化。
- 左侧是一个病例问题卡片。
- 中间是 4 个来源图标：Guideline、PubMed、Drug label、Knowledge base。
- 右侧是 Evidence Brief 输出卡，卡内只画 3 行：Conclusion / Evidence / Caveat。
- 不放大段论文结果。
- 讲解时只沿着箭头讲流程，不展开 Almanac 的完整算法细节。

### 讲述文案

第一个场景是指南与证据检索。

这个场景的工作流比较清楚：先把临床问题结构化，再从指南、PubMed、药品标签和知识库中检索证据，最后输出一个带引用的 Evidence Brief。

Almanac 的研究中，大约 91% 的临床回答引用被评估为正确可信。这个数字说明，RAG 的价值不只是让回答更长，而是让回答更容易追踪来源。

当然，RAG 不等于事实保证。真正决定输出质量的，仍然是检索库质量、更新时间、证据适用人群，以及最后的医生复核。

### 数据来源

- Almanac, NEJM AI 2024 / PMC10857783：正确可信引用 91.11%。
- Ferber et al., NEJM AI 2024：GPT-4 + RAG 用于肿瘤指南检索和比较，作为方法启发；因直接下载受限，不在主画面使用未复核数字。
- Hao et al., npj Digital Medicine 2025：系统综述提示临床决策任务仍需验证。

第二，它的价值不在于一次性给出一个答案，而在于把检索、工具调用、证据引用和人工复核组织成可追踪的工作流。

第三，医学场景不是越自动越好，而是越可复核越安全。

所以，这类系统真正适合先落地的地方，是 Evidence Brief、临床试验预筛和 MDT 会前准备这类证据整理任务。它们离最终治疗决策有一定距离，但又能明显减轻医生在检索、筛选和整理证据上的负担。

这个定位也为最后一页未来展望做了铺垫：如果要把 Agent 真正带入肿瘤临床，重点不是把功能做得更炫，而是把证据链、验证终点和临床责任边界做得更扎实。

### 讲述文案

最后，我把未来展望单独放一页。这里我不想再讲“近期、中期、远期”的实现路线，因为证据链、试验预筛和低风险流程试点在今天的 demo 和已有研究中已经能看到雏形。

我更想强调三个真正面向未来的研究目标。

第一是多模态协同与前瞻验证。肿瘤诊疗天然不是单一文本任务，真实决策会同时依赖指南、病理、影像、分子检测、病程记录和临床试验信息。未来重要的问题不是把这些信息简单拼在一起，而是当不同模态之间出现冲突时，系统能不能识别冲突、解释冲突，并在跨中心前瞻研究中证明它确实有帮助。

第二是多角色 MDT 协作智能体。未来的肿瘤 Agent 不一定是一个模型回答所有问题，而可能更像一个受控的虚拟 MDT：肿瘤内科、外科、放疗、病理、影像、分子诊断和临床试验匹配分别承担不同角色。这里真正需要研究的是，不同角色的证据如何对齐，意见冲突如何呈现给医生，以及最终责任边界如何清楚保留在人类团队手中。

第三是跨中心部署与持续治理。一个系统在单中心、单模板、单病例上表现好，并不代表它能进入真实临床。未来需要关注它在不同医院、不同数据系统、不同指南版本、不同患者亚组中的稳定性；同时还要持续监测隐私保护、模型漂移、偏倚风险和审计记录。

所以，未来重点不是做一个更会回答的模型，而是做一个可验证、可协作、可治理的肿瘤学证据系统。我的汇报到这里，谢谢各位老师。
